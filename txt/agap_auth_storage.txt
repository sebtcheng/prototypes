This page details the plan for **authentication** (Azure Database for PostgreSQL) and **file storage** (Azure Blob Storage) for the AGAP Portal backend, plus stability recommendations for the remaining production concerns. It builds on the React + Node.js migration guide and the Postgres schema build guide.

> Scope note: the **applicant-facing intake flow** (public registration, online application submission, document upload by applicants) is handled by a **separate app and server**. This plan covers the internal HRMO portal only. Where the two systems share the same Azure Postgres database, the tables below are owned/written by this backend; the intake app should treat auth/user tables as read-only unless explicitly coordinated.
> 

## 1. Authentication (Azure Database for PostgreSQL)

All auth data lives in a dedicated table in the same Azure Postgres instance as the rest of the schema. Registration details, usernames, hashed passwords, hashed passcodes, and contact information are stored here.

### 1.1 `users` table

```sql
CREATE TYPE user_role AS ENUM ('hr_officer','approver','admin');
CREATE TYPE user_status AS ENUM ('active','disabled','locked');

CREATE TABLE users (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  username TEXT NOT NULL UNIQUE,
  email TEXT NOT NULL UNIQUE,
  -- Registration / contact details
  full_name TEXT NOT NULL,
  contact_number TEXT,
  office TEXT,               -- e.g. SDO Manila, Division Office
  -- Credentials (NEVER store plaintext)
  password_hash TEXT NOT NULL,   -- Argon2id (preferred) or bcrypt hash
  passcode_hash TEXT,            -- hash of the close-confirmation passcode, same algorithm
  role user_role NOT NULL DEFAULT 'hr_officer',
  status user_status NOT NULL DEFAULT 'active',
  -- Lockout / security bookkeeping
  failed_login_attempts INTEGER NOT NULL DEFAULT 0,
  locked_until TIMESTAMPTZ,
  last_login_at TIMESTAMPTZ,
  password_changed_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX idx_users_username ON users(username);
CREATE INDEX idx_users_email ON users(email);
```

### 1.2 Password & passcode handling (non-negotiable rules)

- **Never store plaintext.** Hash passwords and passcodes with **Argon2id** (preferred) or **bcrypt** (cost ≥ 12). Store only the hash in `password_hash` / `passcode_hash`.
- Verify with a constant-time comparison from the hashing library — never a manual string equality check.
- The existing UI "close-confirmation passcode" becomes a real server-side check against `passcode_hash`, not a hardcoded client value.
- Enforce a password policy at the API layer (min length, complexity) and rehash opportunistically on login if the cost factor is upgraded.
- Track `failed_login_attempts` + `locked_until` to throttle brute force; reset on success.

### 1.3 Sessions

- Use **short-lived JWT access tokens (~15 min) + a rotating refresh token**, or server-side sessions backed by a `sessions` table / Redis. For a single-region internal tool, server-side sessions are simpler and easier to revoke.
- Store refresh/session tokens as **hashes**, never raw, so a DB leak can't be replayed.
- Send tokens in `HttpOnly`, `Secure`, `SameSite=Strict` cookies to avoid XSS token theft.

```sql
CREATE TABLE sessions (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  token_hash TEXT NOT NULL,       -- hash of the session/refresh token
  ip_address INET,
  user_agent TEXT,
  expires_at TIMESTAMPTZ NOT NULL,
  revoked_at TIMESTAMPTZ,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX idx_sessions_user_id ON sessions(user_id);
```

### 1.4 Authorization (RBAC)

- Every mutating endpoint (save review, update status, open/close vacancy, confirm appointment, add vacancy) must check `role` — client-side gating is not security.
- Suggested mapping: `hr_officer` can review/screen; `approver` can confirm appointments and close vacancies; `admin` manages users. Centralize this in a single `requireRole([...])` middleware so rules aren't scattered.
- Attribute every sensitive write to `user_id` in the audit trail (see §3.1).

### 1.5 Forgot / reset password (no email or SMS subscription)

With no paid email/SMS service, the primary recovery path is **admin-mediated reset**, which needs zero external dependencies. This is practical here because the portal is internal with a small, known set of HRMO staff, and an `admin` role already exists to perform resets.

Add a "force change on next login" flag to `users`:

```sql
ALTER TABLE users ADD COLUMN must_change_password BOOLEAN NOT NULL DEFAULT false;
```

And a single-use reset token table (works for both admin-mediated and optional self-service):

```sql
CREATE TABLE password_reset_tokens (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  token_hash TEXT NOT NULL,             -- hash of a random one-time token/temp password, never the raw value
  issued_by UUID REFERENCES users(id),  -- the admin who initiated it (NULL if self-service)
  expires_at TIMESTAMPTZ NOT NULL,      -- short window, e.g. 30-60 min
  used_at TIMESTAMPTZ,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX idx_password_reset_tokens_user_id ON password_reset_tokens(user_id);
```

**Admin-mediated flow (recommended default, no external service):**

1. The user contacts an admin through an existing trusted internal channel (in person, phone, or official DepEd email that lives outside this app).
2. The admin clicks "Reset password" for that account. The system generates a random temporary password (or one-time token), stores **only its hash**, sets a short `expires_at`, and sets `must_change_password = true`.
3. The admin relays the temporary credential to the user out-of-band. It is single-use.
4. On next login the user is **forced to set a new password**; the token is marked `used_at`, `password_changed_at` is updated, and **all existing `sessions` for that user are revoked**.

**Security rules for reset:**

- Store only the hash of the temp password/token; single-use; short expiry.
- On success: revoke all sessions, reset `failed_login_attempts`/`locked_until`.
- Rate-limit reset requests per account/IP.
- Return a generic message that never reveals whether a username/email exists.
- Avoid "security questions" — they're weak, guessable, and add a table of low-value secrets.

**Optional self-service — only if org email already exists (still no new subscription):**

- If DepEd staff already have Microsoft 365 / Outlook mailboxes, the API can send a one-time reset link (the token above) via the **existing organizational email** (Microsoft 365 SMTP or Microsoft Graph) using licenses the org already pays for — no new service. Flow: user enters username → system emails the one-time link to the on-file `email` → user sets a new password. Keep the admin-mediated path as the fallback when email is unavailable.

## 2. File storage (Azure Blob Storage)

Binary files (NOSCA uploads, documentary requirement scans, generated IER/CAR PDFs) go to **Azure Blob Storage**; Postgres only stores **metadata + the blob path**, never the file bytes.

### 2.1 Container layout

```jsx
agap-portal (storage account)
├─ nosca/            # uploaded NOSCA / vacancy source docs
├─ applicant-docs/   # documentary requirement files (per application)
└─ exports/          # generated IER / CAR / appointed-list PDFs
```

Use one container per logical category (or a single private container with virtual folders). All containers are **private** — no anonymous public access.

### 2.2 Metadata table

```sql
CREATE TABLE stored_files (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  container TEXT NOT NULL,          -- e.g. 'applicant-docs'
  blob_path TEXT NOT NULL,          -- full path/key within the container
  original_filename TEXT NOT NULL,
  content_type TEXT NOT NULL,
  size_bytes BIGINT NOT NULL,
  checksum TEXT,                    -- for integrity / dedupe
  uploaded_by UUID REFERENCES users(id),
  application_id UUID REFERENCES applications(id) ON DELETE SET NULL,
  vacancy_id UUID REFERENCES vacancies(id) ON DELETE SET NULL,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX idx_stored_files_application_id ON stored_files(application_id);
CREATE INDEX idx_stored_files_vacancy_id ON stored_files(vacancy_id);
```

### 2.3 Upload & download flow

- **Upload**: prefer **short-lived user-delegation SAS tokens** issued by the API so the browser uploads directly to Blob (keeps large files off the Node process and avoids request-timeout gateway errors). The API records the `stored_files` row once the client confirms the upload. For smaller files or stricter control, proxy through the API instead.
- **Download**: never expose permanent public URLs. Issue a **time-limited read SAS URL** (minutes, not days) on demand, gated by an RBAC check.
- **Auth to Azure**: use **Managed Identity** (via `DefaultAzureCredential`) rather than a hardcoded account key/connection string in app config. Grant the app the `Storage Blob Data Contributor` role.
- **Validation**: enforce allowed MIME types and a max size on both the SAS request and post-upload metadata; reject anything else.
- **Lifecycle**: enable soft delete + versioning on the storage account, and a lifecycle policy to tier/expire old `exports/` artifacts.

## 3. Best-recommendation items (prioritizing stability & avoiding gateway errors/crashes)

The following aren't fully specified elsewhere; these are the recommendations, biased toward uptime and preventing 502/504 gateway errors and process crashes.

### 3.1 Audit trail & concurrency (data integrity)

- Extend `application_history` with an `actor_user_id UUID REFERENCES users(id)` so every status/evaluation change is attributable.
- Add optimistic locking to the review save: include a `version INTEGER` (or compare `updated_at`) on `applications`; reject a save whose version is stale so two HR officers can't silently overwrite each other. Return a clear 409 the UI can handle gracefully rather than crashing.

### 3.2 Connection management (the #1 cause of gateway errors on Azure Postgres)

- Put a **pooler in front of Postgres** (PgBouncer, or Azure's built-in connection pooling) and cap the app's pool size. Azure Postgres has hard connection limits by tier; exhausting them is the most common source of 5xx/timeout cascades.
- Set sane pool + statement timeouts (e.g., `statement_timeout`, pool `connectionTimeoutMillis`) so a slow query fails fast instead of piling up and taking the process down.

### 3.3 Offload heavy work so requests don't time out

- Generate IER/CAR **PDFs asynchronously** (a job queue such as BullMQ, or an Azure Storage Queue + worker) and hand back a download link when ready, rather than blocking an HTTP request on Puppeteer — long synchronous renders are a classic 504 trigger behind Azure App Gateway / Front Door.
- Stream or paginate large list/export responses instead of building giant in-memory payloads.

### 3.4 Resilience & graceful degradation

- Add a **global error-handling middleware** so no unhandled rejection crashes the process; return structured JSON errors.
- Wrap Blob/DB calls with **timeouts + bounded retries with backoff** (retry only idempotent, transient failures); avoid infinite retry storms.
- Run under a **process manager / clustering** (PM2 or container orchestration with restart policy) and implement **graceful shutdown** (drain in-flight requests, close pool) on `SIGTERM` so deploys don't drop connections.
- Expose **`/health` (liveness)** and **`/ready` (readiness — checks DB + Blob reachability)** endpoints for the Azure load balancer/App Service health probes so unhealthy instances are pulled instead of serving errors.

### 3.5 Protect the edge

- **Rate limit** auth and write endpoints (per IP + per user) to prevent brute force and accidental overload.
- Enforce request body size limits and strict CORS (only the portal's origin).

### 3.6 Observability

- Wire **Azure Application Insights** (or OpenTelemetry) for request tracing, dependency timing (Postgres/Blob), and failure alerting so gateway errors are caught early with root-cause context.
- Use **structured logging** with correlation ids; log slow queries.

### 3.7 Notifications

- Keep the in-app `notifications` table as the source of truth; if email delivery is added later, push it through a **queue + worker** (e.g., Azure Communication Services / SendGrid) so a mail-provider outage never blocks or crashes a portal request.
